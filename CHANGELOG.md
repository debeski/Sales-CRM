# Changelog

## v0.2.0
- **Per-employee row-level visibility (multi-user sales model)**: On top of dlux's model-level permissions, records are now filtered by *ownership* so a sales rep sees only their own work, a manager sees the whole store, and a delivery courier sees only their jobs. New `common/access.py` centralizes it: each owned model declares an `OWNER_FIELDS` tuple (ORM lookups pointing at a user) and gains a `view_all_<model>` permission; `apply_ownership(qs, user)` returns the full queryset for a superuser or a holder of `view_all_<model>`, otherwise `filter(OR of owner lookups)`. A model without `OWNER_FIELDS` (the shared product catalog, exchange rates) is never row-filtered. Filtering runs at the **read choke points** (not the model manager, so `invoice.payments.all()` recalcs stay whole): `ScopedListView.get_queryset`, the invoice detail/print/editor/issue/cancel/payment lookups (`_visible_invoices`), and `build_sales_report`.
- **Assignable salesperson on invoices**: `Invoice` gains a `salesperson` FK (defaults to the creating user via `Invoice.save` / the editor) and `OWNER_FIELDS = (salesperson, created_by)`, so ownership survives handoff. The editor's salesperson picker is manager-only (new `assign_salesperson` permission); a rep can't reassign a sale away from themselves. New `view_all_invoice` lifts row-scoping for managers. Added a Salesperson column to `InvoiceTable`.
- **Private customer books**: `Customer` is now owned (`OWNER_FIELDS = (created_by,)`, `view_all_customer`) — a rep only sees/binds/searches their own customers (the invoice editor's customer datalist and `_sync_customer` find-or-create are ownership-filtered). `Payment` (`created_by` OR `invoice__salesperson`, `view_all_payment`) and `finance.CashDeposit` (`created_by`, `view_all_cashdeposit`) are owned too.
- **Delivery board (new `sales.Delivery` model)**: A courier-facing delivery job — optional `invoice` link, `assigned_to` courier, snapshotted recipient/phone/address, a six-state `status` (pending→assigned→out→delivered/failed/cancelled, auto-advances to *assigned* when a courier is set and stamps `delivered_at` on delivery), and `scheduled_date`. `OWNER_FIELDS = (assigned_to, created_by)` so a courier sees only their assigned jobs; `view_all_delivery` + `assign_delivery` for dispatchers. Full surface: `DeliveryListView` (modal CRUD), `DeliveryTable`, `DeliveryFilter`, `DeliveryForm` (courier picker gated by `assign_delivery`), `/sales/deliveries/` route, and EN/AR strings. The auto-discovered sidebar entry appears only for users with `view_delivery`.
- **Dynamic-modal ownership guard**: The project's `scoped_modal_manager`/`scoped_modal_delete` routes resolve objects through dlux's module-level `_scope_filtered_modal_queryset`; `common.access.install_modal_ownership_patch` wraps that single helper (all three edit/view/delete call sites route through it) so a rep can't open another rep's private Customer/Payment/Delivery by guessing its id. Installed on the first `request_started` (not app-init) to avoid a startup DB query from importing `dlux.views`.
- **Role-aware dashboard**: `DashboardView` was `LoginRequiredMixin`-only and showed **global** today/month sales, outstanding balances and recent invoices to *anyone logged in* (a leak). Every panel is now both permission-gated and row-scoped: sales figures/recent-invoices show only under `view_invoice` and cover only the viewer's own sales (whole store for a manager); a new **Deliveries** panel shows under `view_delivery`; low-stock shows only under `catalog.view_product`. So a delivery courier's home page shows deliveries — never sales.
- **Roles command**: `seed_roles` rewritten from one "Sales Staff" group into three — **Sales Manager** (all view_all_* + assign_* + reports + catalog/rate + confirm deposits), **Sales Representative** (own data only, no view_all), **Delivery Courier** (own deliveries + cash collection, no invoices/reports). Run `python manage.py seed_roles`; assign each user one group, keep the owner a superuser.
- **Tests**: +15 (`sales/tests/test_visibility.py`) covering the ownership helper (own/assigned/view_all/superuser/private-customer), list + detail choke points (rep vs manager, cross-rep 404), the modal-lookup patch, delivery assignee scoping + address snapshot, the salesperson default, and the role-aware dashboard context. 46 tests pass. Migrations: `sales/0003` (salesperson + Delivery + Meta perms), `finance/0002` (view_all_cashdeposit).

## v0.1.5
- **Baked DjangoLux bumped to 1.3.2**: `requirements.txt` now pins `django-lux[updater]==1.3.2` (was 1.2.14). The VPS deployment had already inline-updated to 1.3.1, so a rebuilt image baking an older 1.2.x would be **older than the active runtime** and composer's preflight version gate would (correctly) refuse to deploy it. Pinning ≥ the deployed version keeps the baked label (`org.switchlibya.dlux_baked_version`) at or above `active.json`, and picks up the dlux 1.3.2 registry-driven image-update fixes.
- **Fixed `composer-updater --check-image` target**: the registry availability check pointed at the wrong default image (`debeski/decrees:latest`, copied from the decrees template) — corrected to `${WEB_IMAGE:-debeski/sales:latest}` so composer polls this project's own image for a newer digest.

## v0.1.4
- **Baked DjangoLux version label**: `Dockerfile` now takes an `ARG DLUX_BAKED_VERSION` and sets `LABEL org.switchlibya.dlux_baked_version`; `.github/workflows/release.yml` derives it from the pinned `django-lux[updater]==<ver>` in `requirements.txt` (a new "Resolve baked DjangoLux version" step) and passes it via `build-args` to both the smoke-test build and the multi-arch push, so the label ships on the released image. Lets composer's preflight version gate compare a target image's baked framework version against the deployment's active runtime version. Empty locally = gate skipped.
- **Composer-as-updater services**: Added `composer-updater` (image `debeski/composer:1.1.5`, `command: watch`) — a resident, trigger-driven image updater watching `/opt/dlux-runtime/state/image-update-request.json` on the shared `dlux_runtime` volume; on request it runs `composer -uo` (pull → version gate → recreate → health → `post_start` migrator) against the host daemon and publishes `deploy-status.json`. Added `docker-socket-proxy` (tecnativa) as a least-privilege Docker API gateway (containers/images/networks/volumes/exec/POST/info/ping/version only) so the updater never mounts the raw `/var/run/docker.sock`. Version-gate wiring via `COMPOSER_VERSION_LABEL=org.switchlibya.dlux_baked_version` + `COMPOSER_ACTIVE_VERSION_FILE=/opt/dlux-runtime/state/active.json`. The updater is pinned (not `:latest`) to avoid recreating itself mid-update and mounts the project at the host path (`${PWD}:${PWD}`) so compose-in-container resolves the `./media`/`./logs` bind mounts correctly — meaning the stack must be brought up from its root dir.
- **Network topology consolidation**: Replaced the mixed `sales_net` + `sales_internal` + `dlux_update_egress` layout with a clear 3+1 model: `frontend` (Caddy ingress only — a bridge, so the host reaches the app and Caddy can still reach Let's Encrypt/ACME for automatic HTTPS), `egress` (outbound internet: smtp-relay → Gmail, dlux-updater → PyPI, **celery** → rate scrapes cbl.gov.ly/eanlibya.com, composer-updater → Docker Hub), `internal` (no-internet inter-service: db, redis, web, celery, pgadmin, db-backup, Caddy backend), and an isolated `docker_proxy` (composer-updater ↔ docker-socket-proxy only). Same isolation guarantees as before (web still has no internet; celery keeps its egress only for the rate scrape), one obvious egress network instead of two.
- **Dev override + settings**: `compose.dev.yml` now mounts the live `pkg-django-lux/dlux` source into `smtp-relay` too (all app services now run the same editable dlux). `config/settings.py` splits the `dlux_settings` import onto its own line (with a `# noqa` note) so dlux_setup's literal-substring check sees it and stops re-appending its settings block.
- **Portfolio landing — real theme + language toggles**: The static `portfolio/index.html` previously *claimed* EN/AR + light/dark but had no controls (light/dark was OS-`prefers-color-scheme`-only and Arabic was a single hard-coded tagline). Added two header icon toggles: a **theme** switch (sun/moon) that sets an explicit `data-theme` on `<html>` — overriding the OS preference, persisted in `localStorage` (`sl_theme`), applied pre-paint via a tiny `<head>` script to avoid FOUC, and it still tracks live OS changes until the visitor picks one — and a **language** switch (`ع`/`EN`) that swaps the entire page via a `data-i18n`/`data-i18n-html` dictionary (50 keys × EN+AR), flipping `<html lang>`/`dir` for full RTL (Arabic system fonts + reset of Latin letter-spacing), persisted as `sl_lang`. Removed the standalone Arabic `.rtl` tagline (the lede now translates in place). Polish: `scroll-margin-top` for the sticky-header anchor offset, `:focus-visible` rings, theme-change color transitions, `prefers-reduced-motion` guard, toggles stay visible on mobile. Stays a host bind-mount (not baked into the image) by request.

## v0.1.3
- **Split edge into ERP subdomain + static public landing (per-hostname Caddy routing)**: The system is now served on **`erp.switchlibya.ly`** while the apex `switchlibya.ly` / `www.switchlibya.ly` serve a separate **static portfolio** — routed entirely at the Caddy edge (no Django, no dlux public-root feature). The `Caddyfile` moved the full ERP routing into an `(erp_app)` snippet imported by the `{$CADDY_ERP_ADDRESS}` site block, and added a second `{$CADDY_SITE_ADDRESS}` block that `file_server`s `./portfolio` (mounted read-only at `/srv/portfolio`), each hostname getting its own auto Let's Encrypt cert. **Breaking (prod .env):** the ERP host var changed from `NGINX_SERVER_NAME` → **`CADDY_ERP_ADDRESS`**; `CADDY_SITE_ADDRESS` now means the *portfolio* host(s). Defaults keep dev working (`compose.dev.yml` sets `CADDY_ERP_ADDRESS=:80` as the catch-all; the portfolio default `http://portfolio.localhost` never ACMEs and never collides). Updated `x-environment` defaults to `BASE_URL`/`ALLOWED_URLS=https://erp.switchlibya.ly` and `ALLOWED_HOSTS` to include `erp.switchlibya.ly` (apex/www never reach Django, so they're dropped from `ALLOWED_HOSTS`); `CSRF_COOKIE_DOMAIN`/`SESSION_COOKIE_DOMAIN` already track `BASE_HOSTNAME`, scoping ERP cookies to the subdomain. New `portfolio/index.html` is a self-contained, responsive, light/dark landing (smart-locks / access-control theme, EN + AR tagline, discreet "Staff Login → erp.switchlibya.ly"). `docker compose config` + `caddy validate` pass for dev, prod, and default configs. **Requires new DNS A records for `@`/apex and `www`** (only `erp` existed). Docs: `OPERATIONS.md` production-deploy section rewritten for the two-site model.
- **Catalog price fields stay consistent (form ↔ detail)**: The product detail view showed a **0.0 selling price** when only `cost_usd` + `markup_percent` were entered, because `price_usd` was left at 0 and derived only on the fly. `Product.save()` now **persists the derived USD selling price** (mirrors `effective_price_usd`) so detail views and every downstream read see a real number even on API/import/no-JS saves. A new `catalog/js/price_sync.js` (loaded on the Product/Service list pages via a reusable `ScopedListView.extra_scripts` hook, present when the create/edit modal opens) keeps the pricing fields in sync live as you type — edit **markup** → recomputes USD; edit **USD** → recomputes markup; edit **cost** → recomputes USD (keeping markup); the live USD→LYD rate (injected as `data-usd-rate` on the LYD widget from `finance.get_current_rate`) is shown as the LYD field's **placeholder**. Per the "keep LYD live" decision the manual LYD field is left blank unless you deliberately type into it (which then back-fills USD + markup and becomes a real fixed override) — so a blank field still sells at the live black-market rate. Fields are matched via `data-price-{cost,markup,usd,lyd}` hooks; event delegation + a `MutationObserver` handle the AJAX-loaded modal.
- **Detail view shows the live LYD selling price + translated choice labels**: dlux's generic detail builder (`_build_generic_detail_context`) only renders stored model fields (so the *derived* LYD price never appeared) and calls `get_<field>_display()` (which returns the **untranslated** English choice label). Fixed both without forking dlux: `Product`/`Service` expose a computed **"Selling Price (LYD)"** row via `get_modal_context()` (`extra_detail_fields`), rendered by a project override of `templates/dlux/helpers/dynamic_modal_detail.html`; and `Product.get_unit_display` / `Service.get_service_type_display` / `StockMovement.get_movement_type_display` are overridden to return the dlux-translated label (`unit_*`/`svctype_*`/`mtype_*` keys) so the Arabic detail view shows e.g. "علبة" instead of "Box".
- **Translated field help text (project-wide)**: `set_field_attrs` localizes field *labels* but never `help_text`, so form descriptions rendered in English under Arabic. New `common/forms.translate_help_text()` (mirrors `translate_choice_fields`) resolves `help_<model>_<field>` (then generic `help_<field>`) from DLUX_STRINGS and is called from every `ModelForm` (catalog/finance/sales). Added AR/EN `help_*` keys for `product.{sku,markup_percent,price_usd,price_lyd_override}`, `stockmovement.quantity`, and `exchangerate.rate`, and rewrote the Product pricing help text to match the new sync behavior (e.g. "Leave blank to sell at the live rate; enter a value to fix the LYD price").
- **Edge switched from nginx+certbot to Caddy (automatic HTTPS)**: Replaced the `nginx` service and the (never-functional) `certbot` renew-loop sidecar with a single `caddy` service (`Caddyfile`, `caddy:latest`). The old setup deadlocked on deploy — nginx refused to boot because its `ssl_certificate` files didn't exist yet, so port 80 never served the ACME challenge, so certbot could never issue the first cert (and `certbot renew` no-ops with zero certs; the overridden entrypoint also made a one-off `certonly` impossible). Caddy obtains **and renews** Let's Encrypt certs itself on boot with zero one-off commands, which fits the composer "just `./start.sh`" model exactly. Certs/ACME account persist in the new `caddy_data` volume (`caddy_config` for runtime state). The `Caddyfile` ports every rule from the retired `.nginx/default.conf.template` (moved to `.xpose/`): gzip/zstd `encode`, immutable 1-year `Cache-Control` on `/static/*`, `/media/*` with a `404` block on `/media/dlux_backups/*`, `/pgadmin4/*` proxy with `X-Script-Name`, the dlux **maintenance-flag** short-circuit (`file` matcher on `/opt/dlux-runtime/state/maintenance` → `error 503`) and the `handle_errors` maintenance page (`maintenance.html`, served `503` for 502/503/504, mirroring nginx `error_page … =503`), request-body cap, and the `/health` → `web:8000/health/` route. Automatic HTTP→HTTPS redirect is added by Caddy. Fixed a latent bug where port 443 published as `${NGINX_PORT:-443}` (any `NGINX_PORT` collapsed 80 and 443 to one port) — ports are now `${HTTP_PORT:-80}`/`${HTTPS_PORT:-443}`. `CADDY_SITE_ADDRESS` reuses `NGINX_SERVER_NAME`, `CADDY_EMAIL` reuses `DEFAULT_FROM_EMAIL`, `CADDY_MAX_SIZE` (default `10MB`) replaces `NGINX_MAX_SIZE`; healthcheck runs `caddy validate`. `compose.dev.yml` points the override at `caddy` (plain HTTP on host `90`, `CADDY_SITE_ADDRESS=:80`, no ACME). Updated `docs/OPERATIONS.md` with a domain + automatic-HTTPS deploy section. **No ACME email is set** — the first cut defaulted `CADDY_EMAIL` to `webmaster@localhost`, which Let's Encrypt rejects (`invalidContact — Domain name needs at least one dot`), wedging issuance; since a Caddyfile `email`/`tls` directive also can't take an empty value, the directive was dropped entirely. Caddy issues + auto-renews certs with no contact address; expiry notices are opt-in via a `{ email … }` global block in the `Caddyfile`.

## v0.1.0
- **Modular App Architecture**: Scaffolded three DjangoLux apps via `dlux startapp --register` with a strict dependency layering `finance ← catalog ← sales`. Added a non-app `common` package (`common/views.py`, `templates/common/scoped_list.html`) holding the reusable `ScopedListView` so every list page wires `SingleTableMixin` + `FilterView` + `setup_filter_helper` + dynamic-modal "Add" in a few lines.
- **Finance — Global Exchange Rate**: `ExchangeRate` (append-only USD→LYD history; newest row is the live rate, cached under `finance:current_usd_lyd_rate`) with `manage_exchangerate` permission. Central `finance/services.py` (`get_current_rate`, `usd_to_lyd`, `lyd_to_usd`, `quantize_lyd`) is the single source of conversion math; ships a `DEFAULT_RATE` fallback surfaced as "no rate set".
- **Finance — Cash Deposits**: `CashDeposit` (ايداع نقدي) with `pending/confirmed/rejected` workflow, `confirm_cashdeposit` permission, and a `CashDepositReviewView` confirm/reject action.
- **Catalog — Hybrid Pricing**: `Product` priced in USD (`cost_usd` + `markup_percent` → `effective_price_usd`) auto-converted to LYD live, with an optional per-item `price_lyd_override` (the agreed hybrid model). `Service` offerings (installation/maintenance/warranty/delivery) priced in USD, LYD, or "per job". `Category` grouping. Auto-generated `sku` (`P00001`).
- **Catalog — Stock Ledger**: `StockMovement` (in/out/adjustment) is the authoritative inventory ledger; `Product.stock_qty` is excluded from the form and only changes through movements (applied atomically with `F()` on insert). Low-stock detection via `reorder_level`.
- **Sales — Invoices**: `Invoice` with auto number (`INV-000001`), `draft/issued/partial/paid/cancelled` lifecycle, and a **frozen** per-invoice `exchange_rate` plus per-line frozen `unit_price_lyd` so historical totals never shift when the rate moves. `InvoiceItem` supports product, service, and free-text custom lines with snapshot descriptions. `Customer` (optional; walk-ins allowed) and `Payment` (auto-maintains `amount_paid`/status, optional link to a `CashDeposit`).
- **Sales — Invoice Editor & Lifecycle**: Full-page multi-line editor (`InvoiceItemFormSet`) with a vanilla-JS add-row, live line/subtotal totals, and product/service price auto-fill from an embedded price map. `sales/services.py` `issue_invoice` (snapshots the rate record, draws down stock) and `cancel_invoice` (restores stock), both transactional. Detail page with issue/cancel/print/record-payment actions; standalone print-ready HTML invoice (`invoice_print.html`) branded from `APP_CONFIG`.
- **Sales — Dashboard**: Stat cards (sales today/month, outstanding balance, live rate), recent invoices, low-stock list, and pending-deposit alert. Intended as the system home page.
- **Roles**: `python manage.py seed_roles` creates a "Sales Staff" group (technicians/delivery reps) limited to selling, issuing invoices, taking payments, and recording cash deposits — no pricing, rate, or catalog access. Owner runs as a Django superuser.
- **Bilingual (AR/EN)**: `DLUX_STRINGS` label/model/app keys for all three apps so forms, filters and navigation render in Arabic or English via DjangoLux discovery.
- **Security & Audit**: Every domain model extends `ScopedModel` (audit trail, soft-delete, optional scope isolation) and every view is `LoginRequiredMixin` + `PermissionRequiredMixin` (`raise_exception=True`). Custom permissions: `manage_exchangerate`, `confirm_cashdeposit`, `issue_invoice`, `cancel_invoice`.
- **Sales Reporting + XLSX Export**: `sales/reports.py` (`build_sales_report`, `build_sales_report_xlsx`) aggregates issued/partial/paid invoices over a date window — total/collected/outstanding, by-status, daily, top products, top services, and an invoice list. `SalesReportView` (HTML, date-range filter) + `SalesReportExportView` (multi-sheet `.xlsx` via openpyxl, styled headers + money formats). Gated by a new `sales.view_sales_report` permission; the file-download route is marked `sidebar_exclude` so only the report *page* appears in navigation.
- **Negative-Stock Guard**: `issue_invoice` now aggregates demand per tracked product across lines and refuses to issue (transactional `ValidationError` naming each shortage) if any product would go negative — drafts and stock stay untouched on failure.
- **Dev tooling**: `config/settings_dev_sqlite.py` overlay for running migrations/checks locally without Postgres/Redis (dev only).
- **Launch**: Verified the DjangoLux Docker stack (`compose.yml` + `compose.dev.yml`) builds and starts via `./start.sh -d` (composer orchestrator, DEV mode, app on `http://localhost:90`).
- **CI/CD — tag-driven Docker Hub + GitHub Releases**: `.github/workflows/ci.yml` (push/PR to `main`: Django check + tests on `config.settings_dev_sqlite`, plus a Docker build + `scripts/smoke-test.sh` runtime gate) and `.github/workflows/release.yml` (on `v*` tag: verify `tag == VERSION`, smoke-test, multi-arch amd64+arm64 build & push `debeski/sales:<ver>` + `:latest`, then a GitHub Release from the matching `CHANGELOG.md` section via `softprops/action-gh-release`). `scripts/smoke-test.sh` boots the image and applies all migrations on in-image SQLite. Auth via `DOCKERHUB_USERNAME`/`DOCKERHUB_TOKEN` secrets; see `docs/RELEASING.md`. Patterned on the `app-composer` setup.
- **Full AR/EN Localization**: Promoted `common` to an installed (model-less) app hosting `common/translations.py` — shared `label_<col>` table-header keys (DjangoLux's django-tables2 patch localizes headers via these at render time), choice-label keys (`status_*`/`method_*`/`mtype_*`/`svctype_*`/`unit_*`/`source_*`/`kind_*`), and `ui_*`/`page_*`/`pr_*` chrome. Rewrote every hand-written template (invoice list/editor/detail/print, dashboard, report) to pull text from `DLUX_STRINGS`; `ScopedListView` now resolves page titles from `page_*_key`s; tables localize choice cells via a `common.i18n.t` helper and `Invoice.status_label`/`InvoiceItem.kind_label`/`Payment.method_label` properties; the printed invoice flips to `dir="rtl"` in Arabic. Previously only form/filter field labels were translated.
