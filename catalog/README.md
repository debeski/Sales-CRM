# Catalog

This app was generated by `python -m dlux startapp` and follows the recommended DjangoLux layout.

## Files You Already Have

- `models.py` for `ScopedModel`-based data models
- `forms.py` for explicit form classes
- `filters.py` for list filtering
- `tables.py` for `django-tables2` tables
- `views.py` and `urls.py` for app-owned flows
- `translations.py` for `DLUX_STRINGS`
- `templates/catalog/...` for list/form pages

## Dynamic Modal Example

The generated list page intentionally opens **Add Record** through the built-in Dlux dynamic modal manager:

```django
{% url 'modal_manager' 'catalog' 'ExampleRecord' 'new' %}
```

The generated table context menu also uses the same direct modal-manager URL for View and Edit actions. Keep this pattern when you want a quick CRUD modal without writing a custom modal endpoint.

## Recommended Next Steps

1. Rename `ExampleRecord` to your first real model when you are ready.
2. Keep the conventional file names so DjangoLux discovery can find them automatically.
3. Register the app in the project manually, or use `python -m dlux startapp catalog --register`.

## Sections vs Custom Views

- Use `is_section = True` when the model fits the DjangoLux sections workflow and can live in the system-managed CRUD screen.
- Keep custom views when the model needs a project-owned page flow, dashboard, or business process.

## Preferred Helpers and Bases

- Use `dlux/form_base.html` for full-page forms.
- Use `dlux/list_base.html` for list/filter pages.
- Use `set_field_attrs()` in custom forms for the standard widget surface.
- Use `setup_filter_helper()` or `advanced_filter_helper()` for DjangoLux-friendly filters.
